### DATA-LOSS BUG FOUND (reproduced in first run, confirmed by code analysis)

**Scenario 1, first run — kill landed mid-transfer at 7,861,760 bytes (of 209,715,200).**

**Evidence from first run output:**
```
Remote bigfile.bin size after kill:  7,861,760 bytes  (partial)
Remote bigfile.bin mtime after kill: 1782984060  (NEWER than local mtime 1782984017)
rsync push bigfile.bin: signal: killed    → tar fallback started
[caravan killed -9 before tar completed]
Re-sync: "✓ chaos in sync (4 files)"   ← FALSE — files diverged
Remote checksum: 12d5ba6...  ≠  source: 70e369...  (DIVERGED PERMANENTLY)
No conflict backup created.
```

**Root-cause trace:**

1. rsync was killed with `SIGKILL` at 7.8 MB into the 200 MB push.
2. `applyActions` in `remote.go:524-530` caught the rsync error and started the tar fallback.
3. `SIGKILL` on the caravan process also terminated (or partially executed) the tar write.
4. **`sync.go:341-349`**: caravan (or the rescan running just before it died) always rescans both sides after `applyActions` returns, regardless of success. The rescan saw the 7.8 MB partial file on the remote and `buildBase` (`sync.go:778`) recorded it with `rsize=7,861,760` and the partial file's mtime as the authoritative base.
5. `SaveState` persisted this wrong base.
6. Next sync: `Plan()` evaluates `remoteModified = fileModified(r, base.RSize=7.8MB, base.RMtime=...)`. The partial remote file hasn't changed since the base was recorded → `remoteModified=false`. `localModified=false` (local unchanged). Result: **no action, "in sync"**. The remote is permanently diverged with no self-repair path.

**Files:**
- `internal/syncengine/sync.go:341` — rescan always runs after apply (even partial)
- `internal/syncengine/sync.go:343` — `buildBase(newLocal, newRemote)` records whatever was scanned
- `internal/syncengine/sync.go:349` — `SaveState` persists the wrong base
- `internal/syncengine/remote.go:524-530` — rsync fallback to tar; if tar is also killed, partial write is left

**Severity: DATA-LOSS GRADE.** The remote file is permanently 7.8 MB of garbage; caravan never detects the divergence again. No conflict backup is created (the partial file was written by a killed transfer, not treated as a conflict).

**Why the second and third runs passed:** Tailscale LAN throughput on this pair transfers 200 MB in ~3s via rsync. The 0.5s polling interval repeatedly missed the transfer window. The bug is timing-dependent: requires the kill to land after rsync starts writing but before it completes.

**Scenario 3 (SSH mux drop) — PASS with note:**
Caravan correctly falls back from rsync to tar when the SSH ControlMaster is dropped. rsync fails with `exit status 255 / unexpected end of file`; the fallback tar opens a fresh SSH connection (new mux master auto-created), succeeds, and returns exit 0. Caravan logs the rsync error but exits 0 — appropriate because the transfer did succeed via tar. Convergence confirmed correct on re-sync.

**Scenario 4b (missing state) — PASS with note:**  
Deleting the state file on an already-synced pair triggers mass conflict resolution (3 files re-pushed with "conflict: local newer or tie"). This is expected behavior: without a base, both sides look "new" and mtime breaks the tie. Content is preserved; trees converge correctly.

**Scenario 5 (lock contention) — PASS:**  
4 of 5 parallel sync processes correctly printed "skipped: sync already running" and exited cleanly. The flock-based lock (`lock.go`) works. State JSON was valid after parallel fire. One discovered detail during investigation: when caravan is killed with `SIGKILL`, child processes (`tar`, `rsync`) that inherit the flock file descriptor continue holding the lock as orphans until they exit. The test script addresses this with explicit `kill_orphans()` cleanup between scenarios.
